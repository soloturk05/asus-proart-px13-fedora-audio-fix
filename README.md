The symptom (and why it's misleading)
After resuming from suspend, YouTube (and any HTML5 video) just shows a spinner and never starts — often with the message "If playback doesn't begin shortly, try restarting your device."
It looks like a browser / codec / network problem, but:
It happens in both Firefox and Brave/Chromium → not browser-specific.
ping googlevideo.com is clean, large downloads are fast → not network.
Disabling hardware video decode (VA-API) does not help → not the GPU decoder.
Root cause
The TI TAS2783 speaker amplifiers on the AMD SoundWire bus fail to resume after suspend with Fedora's stock kernel. When the audio sink is locked, the browser cannot open an audio output, so media playback stalls forever. The visible symptom is "video won't play," but the actual fault is the internal speaker/audio path.
Proof: temporarily switching the default sink away from the broken TAS2783 output makes the video play but with no sound — confirming the stall comes from the locked audio device, not the network or browser:

The fix has two layers: (A) device-specific userspace quirks (UCM / PipeWire / WirePlumber), and (B) a patched kernel TAS2783 + SoundWire driver set, signed for Secure Boot.
⚠️ Before you start
This is an advanced fix (kernel module compilation + Secure Boot module signing). Take your time.
Commands below assume your username is solo in a couple of absolute paths — replace with your own ($HOME / your user) where you see /home/solo/....
This fix is tied to one specific kernel version. After a kernel update it can break again — see "Surviving kernel updates" at the end.
If you don't want to deal with Secure Boot signing, an alternative is to disable Secure Boot in firmware and skip the MOK steps. (Signing keeps Secure Boot ON, which is recommended.)

Surviving kernel updates (important)
This fix lives in /lib/modules/<KERNEL>/updates/px13-audio, which is not recreated automatically for a new kernel. After any kernel update the problem can return. Quick check after updates:
Bash
If the modules now point back to stock, repeat Steps 3–8 against the new kernel version (re-download source, re-apply patches, rebuild, re-sign with the same MOK key, reinstall). Wrapping this as a DKMS package is recommended so it rebuilds automatically on every kernel update — the MOK key only needs enrolling once.
